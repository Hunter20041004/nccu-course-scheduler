# Sunbreak 高增益可靠性與 UX 改造設計規格

日期：2026-07-16  
狀態：已確認範圍，待依本規格實作  
適用版本：政大 115-1 排課工具完整 Sites 站與 GitHub Pages 作品集入口

## 1. 背景

Sunbreak 已支援政大官方節次、候選課程、官方課程搜尋、課綱、選課條件、實習設定、非同步課程、AI 截圖匯入、AI 推薦方案、使用教學及手機桌布匯出。Chrome 實測顯示主要風險不在功能數量，而在資料語意、班別一致性、AI 方案可信度、雙站狀態分裂、錯誤復原與手機閱讀效率。

本次改造一次涵蓋所有已確認項目，不依優先級刪減範圍；實作仍以垂直切片逐一完成，確保每一步皆有測試保護。

## 2. 目標

1. 官方課程文字不再把授課語言、課程分類、學程標籤或一般說明誤判為必要資格。
2. 多班別、多教師與多討論安排以唯一正式班別資料呈現，課號、教師、時段、課綱與資格不會互相脫節。
3. AI 只負責理解、排序與解釋；任何可見、可預覽或可套用方案都必須先通過確定性排課驗證。
4. GitHub Pages 與完整 Sites 站的角色清楚，提供安全且可理解的本機資料移轉方式。
5. 清空課表與候選課程後可以復原，降低不可逆操作風險。
6. 手機優先提供可快速閱讀的行程模式，並保留完整政大方格課表。
7. 減少候選課程與首頁的操作噪音，同時維持鍵盤、螢幕閱讀器及 44px 觸控範圍。

## 3. 非目標

- 不登入或代替學生操作政大正式選課系統。
- 不以 AI 推測官方未提供的資格、課綱、教師或時段。
- 不建立強制登入或雲端帳號系統。
- 不把 API Key、課表、個人敘述或截圖長期儲存在伺服器。
- 不移除政大完整方格課表、非同步工作區或手機桌布匯出。
- 不全面更換既有雨後日光／輕夢核視覺語言。

## 4. 整體架構

### 4.1 官方資料正規化層

`nccu-course-adapter.mjs` 不再只輸出單一 `restrictionText` 語意。每門官方課程正規化後包含：

```js
{
  eligibilityRules: [],
  scheduleNotes: [],
  deliveryNotes: [],
  examEvents: [],
  programTags: [],
  informationNotes: [],
  source: {
    kind: 'nccu-official',
    url: 'https://...',
    term: '115-1',
    fetchedAt: 'ISO-8601 timestamp'
  }
}
```

分類規則：

- `eligibilityRules`：僅限、限本系、不得、須先修、須具備資格、最低年級或明確身分要求。
- `scheduleNotes`：另約、彈性討論、共同展示、中午討論或一次性會議。
- `deliveryNotes`：英語授課、同步、非同步、遠距平台或實體要求。
- `examEvents`：有日期或可轉換時段的考試、成果展示與共同活動。
- `programTags`：TAICA、學程對應、群組、傳播類、擴大輔系課程等分類。
- `informationNotes`：不屬於以上類別但仍需保留的官方原文。

分類器不得因單獨出現「課程」、「學程」、「輔系」、「群」、「英語授課」等字樣建立必要條件。每筆規則保留原文、來源與 `confidence`；無法確定的內容不得轉成 `required`。

### 4.2 資格三態

資格結果統一為：

- `eligible`：沒有阻擋條件，或使用者已明確符合所有必要條件。
- `blocked`：使用者已明確不符合必要條件。
- `review`：存在尚未回答、資料不足或需人工確認的必要條件。
- `unavailable`：該學期未開課或官方資料確認不可選。

UI 文案：

- `eligible`：條件符合。
- `blocked`：條件不符合，請看詳細。
- `review`：資格待確認，請看詳細。
- `unavailable`：本學期未開課。

選課條件頁只列真正需要學生回答的 `eligibilityRules`。課程分類、英語授課、遠距平台及聯盟說明以資訊卡或標籤呈現，不使用 checkbox。

### 4.3 正式班別模型

每個候選課程可包含多個 `sections`，但必須有且只有一個 `selectedSectionId`。每個 section 是不可拆散的完整資料：

```js
{
  id: '783006001',
  sectionCode: '783006001',
  teacher: '魏綾音',
  credits: 3,
  schedule: { day: 2, periods: ['3', '4', 'C'] },
  meetings: [],
  deliveryMode: 'in-person',
  advisorOptions: [],
  eligibilityRules: [],
  syllabusUrl: 'https://...'
}
```

切換正式班別時，以單一核心函式原子更新有效課號、教師、學分、時段、授課方式、課綱、資格與提醒。班別切換後立即重新計算衝堂、實習天數、資格及所有使用該課程的 AI 預覽。

「人工智慧實務專題」的 783006001／魏綾音班預設顯示週二 34C，並允許使用者在「週二 34C」與「另約討論時間」之間選擇；選擇另約時顯示為時間待確認，不佔固定時段，但產生風險提醒。

### 4.4 確定性方案驗證器

新增純函式方案驗證器，接收候選課程、方案課程 ID、非同步 ID、鎖定課程、使用者資格、實習設定與硬性偏好，輸出：

```js
{
  valid: true,
  courseIds: [],
  asyncCourseIds: [],
  credits: 0,
  internship: {},
  violations: [],
  warnings: []
}
```

硬性違規包括：

- 固定時段或一次性事件衝堂。
- 未保留鎖定課程。
- 包含 `blocked` 或 `unavailable` 課程。
- 使用者明確要求最低學分但方案未達標，且仍有可加入的合格課程。
- 把不可非同步的課程列為非同步。
- 使用者明確要求最低實習天數但方案無法達標。

`review` 或時間未定課程產生 warning，不偽裝成完全確定。AI 回覆先經 schema 與白名單驗證，再由本驗證器修復、補足或丟棄。最終只有 `valid: true` 的方案可以顯示與套用。若無法產生三個互異有效方案，顯示實際可行數量與缺少原因，不用衝堂方案湊數。

### 4.5 本機資料可攜性與雙站角色

完整 Sites 站是唯一功能完整版。GitHub Pages 是作品集與分享安全入口，清楚標示其用途並導向完整站。

兩個 origin 無法直接共享 `localStorage`，因此提供顯式資料移轉：

- 匯出：下載版本化 JSON，包含候選課程、已排課程、鎖定、條件回答、實習設定、自訂行程與必要的資料版本。
- 匯入：預覽將新增、覆蓋或忽略的資料，使用者確認後才寫入。
- JSON 不包含 API Key、AI 個人敘述、截圖或暫時性錯誤資訊。
- 匯入使用 schema 驗證與版本 migration；未知欄位忽略，無效資料拒絕且不破壞現有狀態。
- GitHub Pages 顯示「匯出並前往完整版」；完整站顯示「從舊版匯入」。

完整 Sites 站不得再顯示「請改用完整 Live Demo」的自我導流文案。

### 4.6 可復原的破壞性操作

清空課表與清空候選課程仍需確認，但執行前保存單次 undo snapshot：

- 顯示 15 秒 toast：「已清空課表／候選課程」與「復原」。
- 復原回復候選、課表、鎖定、條件回答與相關提醒的操作前狀態。
- 新的破壞性操作會取代舊 snapshot。
- 重新整理後不保留 undo snapshot，避免把暫存復原狀態當成正式資料版本。
- 清空按鈕從主要 header 降級到「更多操作」選單，保留明確文字與鍵盤操作。

## 5. 介面設計

### 5.1 桌面版

- 保留左課表、右工具的核心結構。
- 首頁 header 只保留「使用教學」、「API Key 狀態」與「更多操作」。
- 課表摘要將實習拆為「已確認 X 天」與「待確認 Y 天」。
- 候選列的主要操作為課程卡本體的加入／移出與「詳細」。
- 「課綱」、「鎖定／解鎖」、「刪除」放進每門課的「更多」選單；桌面與鍵盤皆能操作。
- 詳細頁以區塊呈現正式班別、時間、授課方式、資格、課程分類、重要日期、官方原文與來源，不混成單一備註。

### 5.2 手機版

手機「課表」分頁新增兩種檢視：

- `行程`：預設。依週一至週日顯示有課或實習的日卡，按時間排序；非同步與時間未定另列。
- `方格`：保留完整政大官方方格並允許水平捲動。

要求：

- 390px 寬度不產生頁面級水平溢出。
- 工具 tabs 的可點擊高度至少 44px。
- 候選操作不穿模，更多選單不超出 viewport。
- 行程模式能直接看到課名、節次／時間、課號、授課方式、鎖定及衝堂狀態。
- 使用者切換檢視的選擇保存在同一瀏覽器。

### 5.3 視覺與動態

- 延續米白、暖黃、霧藍與淡紫；不加入薄荷綠或亮天藍。
- 以資訊層級、間距、邊框與少量色彩區分狀態，不增加裝飾性卡片堆疊。
- toast、popover 與檢視切換使用 120–180ms 動畫，尊重 `prefers-reduced-motion`。
- 任何狀態不能只靠顏色傳達。

## 6. 錯誤處理與可觀測性

- 官方資料查詢回應經 schema 驗證；欄位缺失時保留原始可用資訊並標示待確認。
- 官方來源不可用時顯示可重試訊息、最後成功更新時間與不含個資的 request ID。
- AI 錯誤區分未設定 Key、Key 無效、額度／頻率限制、逾時、上游暫時錯誤與輸出不可驗證。
- UI 提供重試，不清空已填表單或現有課表。
- Worker 錯誤回應使用穩定的 `code`、`message`、`retryable` 與 `requestId`，不回傳 Key、prompt、截圖或供應商原始堆疊。
- 正式站顯示官方資料學期及最後更新時間，避免把快取資料誤認為即時選課結果。

## 7. 儲存與 migration

- 儲存 schema 版本遞增。
- migration 將舊 `restrictionText` 與 `eligibilityRules` 重新分類，移除資訊性誤判但保留官方原文。
- 舊 variant／advisor 欄位轉成 sections 與 selectedSectionId；無法唯一對應時保留原課程並標示待確認。
- migration 必須具冪等性；重複載入不持續改寫資料。
- 不變更既有使用者的候選課程、鎖定、課表或自訂行程，除非該欄位可確定為舊版錯誤衍生資料。

## 8. TDD 實作順序

雖然產品範圍全部納入，程式仍依下列垂直切片逐一 Red → Green → Refactor，一次只新增一個失敗測試：

1. 官方備註分類：先重現英語授課、傳播類、群 C 與 TAICA 說明誤判，再加入正規化分類器。
2. 資格三態：先重現未回答條件被當作符合或阻擋，再加入 `review` 評估與 UI 文案。
3. 正式班別：先重現人工智慧實務專題課號、教師與時段不同步，再加入原子 section 選擇。
4. AI 方案驗證：依序測衝堂、最低學分、非同步、鎖定、資格與實習硬性條件；每條規則各自一個循環。
5. 有效方案輸出：先重現用衝堂方案湊滿三個，再改為只回傳可行方案及不足原因。
6. 資料匯出：先測 JSON 不含 Key 與個人敘述，再加入版本化匯出。
7. 資料匯入：依序測合法預覽、無效拒絕、migration 與確認寫入。
8. 清空復原：先測清空後狀態，再測 15 秒內復原與新操作覆蓋 snapshot。
9. 候選更多選單：先測鍵盤與 44px 操作，再調整渲染與樣式。
10. 手機行程模式：先測資料排序與分組，再測 390px 無頁面溢出與檢視持久化。
11. 正式站文案與入口：先測不同部署角色，再移除完整站的錯誤導流並加入移轉入口。
12. 真實邊界與回歸：政大 live contract、Worker API contract、建置、lint、Chrome 桌機與手機完整流程。

## 9. 真實邊界驗證

- 政大 live contract 至少驗證一門有固定時段課程、一門含資訊性備註課程及一門明確資格限制課程。
- 官方課綱連結需實際回傳對應 115-1 課名、課號與時段。
- Worker contract 驗證 request ID、錯誤 shape、no-store 與敏感資料不外洩。
- Chrome 正式站驗證官方搜尋、加入候選、切換班別、資格回答、AI 方案、實習、清空復原、JSON 移轉、手機行程與完整方格。
- 瀏覽器測試使用測試用空白工作區，不改動使用者目前保存的個人課表。

## 10. 完成條件

- 「英語授課」、「傳播類課程」、「群 C 課程」、「TAICA／學程說明」不再建立必要資格 checkbox。
- 「僅限歐文系及雙主修」、「須先修」、「限本系」等明確限制仍正確建立條件。
- 未回答的必要條件顯示待確認，不誤稱符合。
- 人工智慧實務專題的選定班別、課號、教師、時間、課綱與資格一致；魏綾音班可選週二 34C 或另約。
- AI 不顯示或套用任何衝堂、資格不符、遺失鎖定課或不合法非同步方案。
- 使用者要求最低學分／實習天數時，方案符合要求；不可達成時清楚說明且不偽造可行方案。
- GitHub Pages 與完整 Sites 站角色清楚，資料可以版本化 JSON 安全移轉。
- 清空後 15 秒內可復原。
- 手機預設行程模式可快速看完一週，仍可切回完整政大方格。
- 390px 寬度無頁面水平溢出，主要觸控目標至少 44px。
- API Key 與個人敘述不進入 localStorage、匯出檔、日誌或錯誤回應。
- 完整 unit、rendered HTML、build、lint、政大 live contract 與 Chrome 正式站流程通過。

## 11. 設計自我檢查

- 無 TBD、TODO 或待決欄位。
- 資料模型、UI 狀態、migration、AI 驗證及完成條件使用相同名稱與語意。
- 正確性、可復原性、手機 UX 與部署一致性皆有獨立且可驗證的完成條件。
- 未把跨 origin localStorage 描述成可直接同步；資料移轉明確採使用者控制的版本化 JSON。
- AI 與官方資料邊界均有真實契約測試，不以 mock 取代。
- 本規格不要求登入、雲端同步或自動正式選課，範圍保持在既有產品能力的高增益改造。
