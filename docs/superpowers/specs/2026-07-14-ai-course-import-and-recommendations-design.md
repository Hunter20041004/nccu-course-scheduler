# AI 課程匯入與三方案排課設計

日期：2026-07-14  
狀態：待使用者審閱  
網站：NCCU Course Planner 私人版

## 1. 目標

本次修改在既有政大排課系統加入兩組能力：

1. 所有候選課程都有一致的「詳細」、「鎖定」、「刪除」控制；任何課程都能加入課表並鎖定。
2. 使用者上傳政大課程查詢網站的追蹤清單截圖後，系統用 AI 辨識課程、核對現有資料及官方資訊，再匯入候選課程。
3. 使用者填寫自我介紹、未來方向、本學期目標與排課需求後，系統產生三個可直接套用的推薦課表。
4. AI 套用方案時保留使用者已鎖定的課程，並由本機排課規則重新檢查撞堂、資格、實習時段與特殊分組時段。
5. Groq API 金鑰只存在伺服器端加密環境變數，不進入前端、版本控制、日誌或錯誤訊息。

## 2. 非目標

- 不自動登入政大選課系統，也不代替使用者送出正式選課。
- 不把 AI 辨識結果直接視為官方事實；無法核對的課程必須進入「待確認」。
- 不以 AI 回覆取代現有的資格、衝堂與實習可行性計算。
- 不在伺服器長期保存截圖、個人敘述或 AI 請求內容。
- 本階段不建立多使用者帳號、雲端同步或歷史版本管理。

## 3. 現況與修改範圍

目前網站是單頁私人排課工具，資料保存在瀏覽器 `localStorage`，內建 115-1 已確認開課的候選課程、政大節次方格、實習時段、手動行程、鎖定課程與快速方案。截圖功能目前只有本機預覽及提示文字，未連接 AI。候選課程的操作按鈕依課程類型不同，並非每門都顯示完整操作。

本次保留現有資料模型與雨後日光／輕夢核視覺，不重做整個應用。AI 由既有 Sites Worker 增加私有 API 路由承接，前端只取得經過驗證的結構化結果。

## 4. 整體架構

### 4.1 Worker 路由

- `GET /`：回傳現有單頁網站。
- `POST /api/ai/import-courses`：接收單張追蹤清單截圖，辨識、比對及回傳可匯入與待確認課程。
- `POST /api/ai/recommend-plans`：接收使用者需求和目前候選課程，回傳三個推薦方案。
- 其他路徑回傳結構化 `404`。

AI API 使用伺服器端 `fetch` 呼叫 Groq 的 OpenAI 相容 Chat Completions 端點，不把 Groq SDK 或金鑰打包到瀏覽器。模型固定為 `qwen/qwen3.6-27b`，避免由前端任意改模型或端點。

### 4.2 環境變數

- `GROQ_API_KEY`：Sites 的伺服器端加密環境變數。
- 金鑰不得出現在原始碼、產出 HTML、測試快照、Git commit、用戶端網路回應或 console。
- 若未設定金鑰，API 回傳可理解的設定錯誤，既有手動排課功能仍可使用。

### 4.3 官方資料核對策略

政大課程查詢支援以課號、課名、教師、學分、開課單位與時間等欄位查詢；追蹤清單也能由課程代碼進入課程資訊。由於不依賴未公開且可能變動的內部端點，本系統採用可替換的官方查詢介面：

1. 優先和網站內建、已確認於 115-1 開課的課程做精確比對。
2. 找不到時，以學期和課程代碼查詢政大公開課程頁；沒有代碼才以正規化課名和教師比對。
3. 只有唯一且可驗證的開課結果才標記為已核對。
4. 官方頁面暫時不可用、結果不唯一、查無本學期開課或資料不完整時，放到「待確認」，不偽裝成已驗證課程。
5. 官方查詢實作封裝在 adapter，之後可在不更動前端流程的情況下更新來源。

## 5. 候選課程操作

每門候選課程固定顯示三個操作：

- **詳細**：展開學分、教師、資格、授課方式、上課時段、特殊分組與資料來源。
- **鎖定／解鎖**：未在課表上的課程按下「鎖定」時，先加入課表再鎖定；已在課表上的課程切換鎖定狀態。鎖定後，一鍵清空、快速方案及 AI 方案都不得移除它。
- **刪除**：所有課程都可從候選清單刪除，並同步自課表與鎖定集合移除。刪除三門核心課時會額外提醒「這是你標記為一定要修的課程」，但最後決定權仍在使用者；重新載入預設候選課程可復原它們。

點擊課程卡片本體維持加入／移出課表的行為；若課程已鎖定，移除操作會被阻止並告知須先解鎖。三個操作按鈕不觸發卡片本體的切換事件。

## 6. AI 截圖匯入

### 6.1 前端流程

1. 使用者選擇或拖放一張 PNG、JPEG 或 WebP 截圖。
2. 前端先檢查格式與大小；限制單張，原始檔建議不超過 3 MB，編碼後請求不得超過 Groq 的 4 MB base64 限制。
3. 顯示縮圖、隱私提示與「開始辨識」按鈕。隱私提示說明截圖會傳給 Groq 處理、網站不長期保存。
4. 辨識時顯示進度狀態並避免重複送出；完成後分成「已匯入」、「已存在」與「待確認」。
5. 已核對課程直接合併進候選課程；相同課程不重複建立。
6. 待確認項目顯示辨識到的課名、代碼、教師、時間與原因，讓使用者後續編輯、確認或刪除；在確認前不參與 AI 推薦。

### 6.2 API 契約

請求：

```json
{
  "imageDataUrl": "data:image/png;base64,...",
  "term": "115-1"
}
```

Groq 僅負責把不可信的圖片內容轉為結構化資料，預期結構為：

```json
{
  "recognizedCourses": [
    {
      "courseCode": "",
      "title": "",
      "teacher": "",
      "credits": null,
      "scheduleText": "",
      "confidence": 0
    }
  ]
}
```

伺服器驗證 MIME、base64、欄位型別、數量上限與 JSON 結構，最多接受 50 筆辨識結果。回應：

```json
{
  "importedCourses": [],
  "duplicates": [],
  "pendingCourses": [],
  "warnings": []
}
```

比對優先序為「完整課程／班別代碼」→「正規化課名＋教師」→「官方查詢」。課名相似但代碼或教師衝突時不得自動合併。

## 7. AI 三方案排課

### 7.1 輸入介面

AI 排課顧問放在既有快速方案附近，以漸進展開表單降低畫面密度。欄位包含：

- 自我介紹：背景、系級、能力與目前狀態。
- 想做的事：本學期想投入的領域或專案。
- 未來方向：實習、研究、創業、職涯或其他方向。
- 本學期目標：希望完成的具體成果。
- 排課偏好：課量、早晚課、非同步遠距偏好、可接受的密集程度及補充需求。
- 現有實習設定：直接讀取網站已設定的天數、固定／彈性模式與每日時段，不要求重填。

欄位會限制合理長度並在送出前提示個人敘述會傳給 Groq 處理、不會長期保存。

### 7.2 API 契約

請求以精簡且結構化的候選課程資料為主：

```json
{
  "profileText": "",
  "desiredActivities": "",
  "futureDirection": "",
  "semesterGoals": "",
  "preferences": "",
  "internshipSettings": {},
  "courses": [],
  "selectedCourseIds": [],
  "lockedCourseIds": []
}
```

Groq 回覆 JSON：

```json
{
  "summary": "",
  "plans": [
    {
      "id": "balanced",
      "title": "",
      "reason": "",
      "courseIds": [],
      "attendance": "",
      "tradeoffs": []
    }
  ]
}
```

伺服器只允許回傳請求內存在、已確認開課且資格符合的課程 ID；拒絕 AI 虛構的課號，並要求恰好三個互有差異的方案。前端收到後再次使用既有排課引擎計算：

- 同時段衝堂與特殊分組衝堂。
- 實習固定／彈性天數是否可達成。
- 非同步遠距課的時間占用方式。
- 課程資格與未解決的授課選項。
- 鎖定課程是否完整保留。

若 AI 方案包含規則問題，不直接套用為看似正常的課表；方案卡會清楚標示警告及可行性。伺服器逾時目標為 45 秒，前端提供失敗後重試，不影響原有課表。

### 7.3 套用推薦方案

三張精簡方案卡顯示定位、課程數、學分、可留實習時段、選課理由與主要取捨。按下「套用此方案」時：

1. 以方案課程加上所有已鎖定課程形成新課表。
2. 不自動新增新的鎖定狀態。
3. 保留候選清單、手動行程、實習設定與尚未確認的匯入項目。
4. 對需要選擇教師／分組時段的課程保留既有選項；未選者提示使用者完成設定。
5. 寫入 `localStorage` 後立即重新渲染，並提供成功狀態訊息。

## 8. 介面與可用性

- 延續雨後日光的米白、暖黃、霧藍與淡紫色系，不導入薄荷綠或亮天藍。
- AI 功能分成「截圖匯入」與「AI 排課顧問」兩區，預設只展開當前操作，避免壓縮一眼可讀的課表及候選清單。
- 所有 loading、成功、警告與錯誤訊息使用 `aria-live`；按鈕有 disabled、focus-visible 及鍵盤操作狀態。
- 詳細內容使用可存取的 disclosure；鎖定與刪除有明確文字而非只靠圖示或顏色。
- 行動版維持課表優先，AI 表單與方案改為單欄；桌面版方案三欄或可橫向比較。
- 刪除仍需確認，避免誤刪 AI 匯入後已調整的課程。

## 9. 安全、隱私與錯誤處理

- 截圖和個人敘述只在請求期間處理，不寫入伺服器儲存或分析日誌。
- 圖片文字一律視為不可信資料；system prompt 明確禁止遵循圖片內指令，只抽取課程欄位。
- AI 回覆必須通過 schema 與課程白名單驗證，不能觸發工具、任意網址、HTML 或程式碼執行。
- 伺服器限制 Content-Type、請求大小、文字長度、圖片數與輸出筆數。
- 錯誤映射：圖片過大為 `413`、格式／欄位錯誤為 `400`、頻率限制為 `429`、上游驗證或暫時錯誤為 `502`、逾時為 `504`、伺服器未設定金鑰為不洩漏細節的 `503`。
- UI 不顯示上游原始錯誤、prompt 或金鑰，只顯示可行的下一步。
- 網站目前為私人存取，但仍不能把私人部署當作唯一安全邊界。

## 10. TDD 與真實邊界驗證

依垂直切片逐次執行 Red → Green → Refactor；一次只新增一個失敗測試及對應最小實作：

1. **課程操作切片**：先測未選課按「鎖定」會加入且鎖定，再實作；下一輪測所有課程顯示三操作，再實作；下一輪測刪除同步清理課表與鎖定，再實作。
2. **匯入 API 基礎切片**：先測錯誤圖片輸入被拒絕，再加最小驗證；下一輪測合法圖片轉成固定 Groq 請求，再加 client；下一輪測 schema 解析，再加驗證；下一輪測重複、已核對與待確認分流，再加比對器。
3. **匯入 UI 切片**：依序測選圖預覽、送出狀態、結果合併、待確認顯示及重試，每次只完成一個可操作流程。
4. **推薦 API 切片**：依序測輸入驗證、Groq 結構化請求、三方案 schema、未知課程 ID 過濾及錯誤映射。
5. **推薦 UI 切片**：依序測表單送出、三方案呈現、規則警告、套用方案保留鎖定課程與持久化。
6. **重構**：抽出共用 AI client、schema validator、API error 與狀態元件；每次重構後跑完整測試。

跨服務邊界不可只靠 mock：

- 單元測試以注入的 fake fetch 驗證請求組裝、錯誤轉換與規則，但不能作為 Groq 契約證明。
- 加入以 `GROQ_API_KEY` 條件啟用的 Groq live contract test，使用極小的文字與圖片 fixture，實際驗證模型接受的 message/image 格式、JSON 回應形狀與錯誤契約；測試不得印出金鑰或完整個資。
- 官方課程 adapter 使用固定公開課程頁 fixture 做解析測試，並以一筆已知 115-1 課程做 live contract check；來源不可用時確認會降級為待確認。
- 部署後使用 Chrome 實測完整私人網站：三按鈕、鎖定任意課程、截圖辨識與匯入、重複項目、待確認、AI 三方案、方案套用、保留鎖定、一鍵清空、手動行程、實習設定及重新整理持久化。

## 11. 部署與金鑰處理

1. 完成測試後，將使用者提供的測試金鑰設定為 Sites 加密環境變數 `GROQ_API_KEY`；不寫入 `.env` 或 repo。
2. 部署 Worker 與靜態網站到既有私人 Sites 專案。
3. 跑真實 Groq contract test 與 Chrome E2E。
4. 若任何測試或部署輸出意外包含金鑰，立即停止並要求撤銷、輪替金鑰。
5. 因金鑰曾透過聊天傳遞，完成測試後建議使用者在 Groq 控制台輪替，再更新 Sites secret。

## 12. 驗收條件

- 每門候選課程均可看到「詳細」、「鎖定」、「刪除」；任何課程都能加入並鎖定，核心課不會預設鎖定。
- 任一課程刪除後不殘留於課表或鎖定集合；刪除核心課前有額外提醒。
- 支援單張政大追蹤清單截圖，已核對課程會加入候選清單，重複課程不重建，模糊或未開課項目不會冒充已核對課程。
- AI 表單能產生三個內容不同且只引用現有合格課程的方案。
- 套用任一方案都保留所有鎖定課程，且衝堂、實習與資格檢查以本機規則結果為準。
- API 金鑰不出現在前端 bundle、Git、console、network response 或測試輸出。
- 所有新增單元／整合測試通過，Groq 與官方查詢真實契約測試通過或有明確降級證據，Chrome 正式站完整流程通過。

## 13. 設計自我檢查

- AI 只做理解與排序，確定性排課規則仍由程式掌握，避免模型產生看似合理但衝堂的方案。
- 「自動匯入」與「官方核對」沒有混為一談；不可驗證的資料有清楚待確認狀態。
- 鎖定行為對所有課程一致，並明確定義未選課程的鎖定會先加入課表。
- 三個核心課仍可自由不選、不鎖或刪除；刪除前的額外提醒兼顧「一定要修」的原始目標與使用者最終控制權。
- 安全設計涵蓋密鑰、圖片、個資、prompt injection、上游錯誤與輸出 schema。
- 外部 API 與官方資料來源均有真實邊界驗證，不以純 mock 測試取代。
